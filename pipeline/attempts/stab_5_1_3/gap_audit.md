# Gap audit: \([[5,1,3]]\) Five-qubit perfect code

## Repo gaps (this code surfaces / requires new abstractions)

This is the **first non-CSS code** in the repository, so it surfaces
gaps where the existing infrastructure is currently CSS-specific. Each
gap below is either (a) a new general-form helper that should be added
in this Stage-4 work and reused by future non-CSS codes, or (b) a
workaround we can do inline at the call site.

### Gap 1 (HIGH PRIORITY): no general "phase-0 + independent ⇒ −I ∉ closure" lemma

**Problem.** The only existing way to discharge `−I ∉ Subgroup.closure
generators` is `CSS.negIdentity_not_mem_closure_union`
(`Core/CSSNoNegI.lean:29`), which requires the generators to be
partitioned into Z-type and X-type subsets that commute generatorwise.
None of the five-qubit code's generators are Z-type or X-type.

**Proposed fix.** Add to `Core/SubgroupLemmas.lean` (or a new
`Core/NoNegI.lean` next to `Core/CSSNoNegI.lean`):

```lean
/-- If `L` is a list of phase-0 Pauli group elements whose check-matrix
rows are linearly independent over `ZMod 2`, then `−I` is not in the
subgroup generated by `L`.

This is the general-form replacement for
`CSS.negIdentity_not_mem_closure_union`, which only works when the
generator set splits into commuting Z-type / X-type subsets.

Proof outline:
1. Every Pauli group element of phase 0 squares to 1 (since `P · P = I`
   on each qubit, and there's no phase contribution at phase 0).
2. So `Subgroup.closure (listToSet L)` is an elementary abelian
   2-group; every element factors as `∏ aᵢ · L.get i` with `aᵢ ∈ {0, 1}`.
3. The operator-part of such a product is `∑ aᵢ · symplectic(L.get i)`
   (mod 2).
4. If `−I ∈ closure`, then `(−I).operators = identity n` is in the
   symplectic span; by row independence, all `aᵢ = 0`, hence the
   product is `1`, not `−I`.
5. Contradiction with `negIdentity_ne_one`. -/
theorem negIdentity_not_mem_of_independent_phase_zero
    {n : ℕ} (L : List (NQubitPauliGroupElement n))
    (hPhase : NQubitPauliGroupElement.AllPhaseZero L)
    (hIndep : NQubitPauliGroupElement.rowsLinearIndependent L) :
    StabilizerGroup.negIdentity n ∉
      Subgroup.closure (NQubitPauliGroupElement.listToSet L) := by
  sorry
```

**Cost estimate.** ~40 LoC in the core lemma (Stage-4 work for this
code); afterwards, every non-CSS code calls this in 1-2 lines.

**Alternative (lower-cost fallback).** Inline the argument in
`FiveQubit_5_1_3.lean` as a private lemma. Cost ≈ 30-50 LoC at the
call site, but the abstraction is lost. Stage-3 review should decide
which path to take — the general lemma is preferred since it amortizes
across all future non-CSS codes.

**Stage-4 trigger.** This lemma is the **gating prerequisite** for
`negIdentity_not_mem` in `FiveQubit_5_1_3.lean`. Stage-4 must close
this lemma (or the inline equivalent) before any other proof in §6+
can compile.

### Gap 2 (MEDIUM PRIORITY): no weight-2 anti-witness helper

**Problem.** `Core/CSSDistance.lean:31` provides
`no_weight_one_mem_centralizer_of_anticommute_witness` — the weight-1
analog of what we need for the lower-bound part of the distance proof.
The [[4,2,2]] code uses it with d = 2, so it only needs weight-1.
For d = 3, we additionally need a weight-2 version, which does not
exist.

**Proposed fix.** Add to `Core/CSSDistance.lean` (despite the file
name, the existing lemma is fully general; same will hold here):

```lean
/-- The weight-2 group element at qubits `i ≠ j` with single-qubit
Paulis `P` and `Q` at those positions (phase 0). -/
def weightTwoAt (i j : Fin n) (P Q : PauliOperator) :
    NQubitPauliGroupElement n :=
  ofOperator (fun k => if k = i then P else if k = j then Q else .I)

/-- If for every pair of qubits and every pair of non-identity
single-qubit Paulis there exists a generator that anticommutes with
the corresponding weight-2 element, then no weight-2 Pauli is in the
centralizer of the stabilizer group generated by `genSet`. -/
theorem no_weight_two_mem_centralizer_of_anticommute_witness
    (S : StabilizerGroup n)
    (genSet : Set (NQubitPauliGroupElement n))
    (h_closure : S.toSubgroup = Subgroup.closure genSet)
    (h_anticomm : ∀ i j : Fin n, i ≠ j →
      ∀ P Q : PauliOperator, P ≠ .I → Q ≠ .I →
        ∃ g ∈ genSet, Anticommute (weightTwoAt i j P Q) g)
    (g : NQubitPauliGroupElement n) (hg : weight g = 2)
    (h_cent : g ∈ centralizer S) : False := by
  sorry
```

**Cost estimate.** ~50 LoC in `Core/CSSDistance.lean`, mirroring the
weight-1 proof. The proof structure is: from `weight g = 2`, extract
the two-element support; use the `h_anticomm` witness at those two
qubits with the local Paulis; derive an anticommutation that
contradicts `g ∈ centralizer`. ~95% structural copy of
`no_weight_one_mem_centralizer_of_anticommute_witness`.

**Alternative (lower-cost fallback).** Try `native_decide` on the
weight-2 sub-case of the distance proof directly. If it closes the
sub-goal in reasonable time, skip the general lemma. Stage-3 review
should record this fallback option.

**Stage-4 trigger.** Required only if the optimistic
`native_decide` on full `HasCodeDistance` fails or is too slow. Try
the easy path first.

### Gap 3 (LOW PRIORITY): no "mul by stabilizer preserves non-triviality" lemma

**Problem.** The distance witness for [[5,1,3]] is
`logicalX * g₁ = -IYYIX` (weight 3), not `logicalX` itself (which has
weight 5). Proving
`IsNontrivialLogicalOperator (logicalX * g₁) stabilizerGroup`
from
`IsNontrivialLogicalOperator logicalX stabilizerGroup`
requires a helper "multiplying by a stabilizer element preserves
non-triviality":

```lean
lemma IsNontrivialLogicalOperator_mul_mem_subgroup
    {S : StabilizerGroup n} {g : NQubitPauliGroupElement n}
    (hg : IsNontrivialLogicalOperator g S)
    (s : NQubitPauliGroupElement n) (hs : s ∈ S.toSubgroup) :
    IsNontrivialLogicalOperator (g * s) S
```

**Search results.**

- `QEC/Stabilizer/Framework/Core/Logical/LogicalOperators.lean` — no such lemma found.
- `QEC/Stabilizer/Framework/Core/Logical/LogicalOperatorCoset.lean` — file exists; this
  is the most likely home if it's already there. Stage-4 should
  `grep "IsNontrivialLogicalOperator" Core/LogicalOperatorCoset.lean`
  before writing it.
- `QEC/Stabilizer/Framework/Core/Logical/LogicalOperators.lean:327-332` —
  `xOp_IsNontrivialLogicalOperator` exists at the `LogicalQubitOps`
  level (logical-X is nontrivial), but no general
  "multiply-by-stab" lemma.

**Proposed fix.** Add to `Core/LogicalOperatorCoset.lean`:

```lean
theorem IsNontrivialLogicalOperator.mul_mem_subgroup
    {S : StabilizerGroup n} {g : NQubitPauliGroupElement n}
    (hg : IsNontrivialLogicalOperator g S)
    {s : NQubitPauliGroupElement n} (hs : s ∈ S.toSubgroup) :
    IsNontrivialLogicalOperator (g * s) S := by
  refine ⟨?_, ?_, ?_⟩
  · -- g * s ∈ centralizer (since centralizer is a subgroup and both g
    --   and s are in it; s ∈ centralizer because the stabilizer is
    --   self-centralizing... wait, is it? S ⊆ centralizer S iff S is
    --   abelian, which it is.)
    sorry
  · -- g * s ∉ S: if g * s ∈ S, then g = (g * s) * s⁻¹ ∈ S, contradiction.
    sorry
  · -- ∀ s' ∈ S, s'.operators ≠ (g * s).operators:
    -- if s'.operators = (g * s).operators, then
    -- (s' * s⁻¹).operators = g.operators, and s' * s⁻¹ ∈ S.
    -- But hypothesis says no element of S has operators = g.operators.
    sorry
```

**Cost estimate.** ~20 LoC.

**Alternative.** Define `logicalX_weight3` directly as a `phasePower
= 2`, `operators = IYYIX` element and prove
`IsNontrivialLogicalOperator` for it from scratch (centralizer-check
via `pauli_comm_even_anticommutes` against the 4 generators;
not-in-subgroup via the symplectic span argument; third condition by
explicit search over the 16-element subgroup). ~80 LoC. The general
helper is the cleaner path.

**Stage-4 trigger.** Only if the `native_decide` distance proof fails.

### Gap 4 (NICE-TO-HAVE): no `weightTwoAt`, `weightThreeAt`, ... helpers

**Problem.** `Core/CSSDistance.lean:25` defines `weightOneAt`, but no
analog for weight 2 or higher. Each new distance proof that needs
manual enumeration above weight 1 currently has to inline these
definitions.

**Proposed fix.** Add `weightTwoAt`, `weightThreeAt`, ... to
`Core/CSSDistance.lean` alongside `weightOneAt`. (Or, better, a single
parametric `weightAt (i : Finset (Fin n)) (P : Fin n → PauliOperator)`
with a default `I` outside `i` — but that's a refactor of the existing
`weightOneAt`, which has stale call sites.)

**Stage-4 trigger.** Not needed for this code unless the manual
weight-2 enumeration is taken (and even then, the local one-liner
`weightTwoAt` definition above suffices).
