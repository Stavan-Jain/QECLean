# Plan: Approach B — Lifted-Product / Tillich–Zémor distance bound

## Goal

Establish a lower bound on `grossHomologicalCode.distance` via the
Lin–Pryadko (2306.16400) Statement 12 / Tillich–Zémor analog for 2BGA
codes:

```
d(LP[a, b]) ≥ d_0 := ⌈ min(d_A^⊥, d_B^⊥) / c ⌉
```

where:
- `a = x^3 + y + y^2`, `b = y^3 + x + x^2` ∈ F_2[Z_12 × Z_6]
- `A = L(a)`, `B = L(b)` are the convolution-by-`a`, convolution-by-`b`
  endomorphisms on F_2[G], viewed as 72 × 72 matrices.
- `d_A^⊥ := d({f ∈ F_2[G] : a · f = 0})` is the classical distance of
  the (linear, F_2) code whose parity-check matrix is `L(a)`.
- `c := |G_a ∩ G_b|` where `G_a`, `G_b` are the subgroups generated by
  the supports of `a`, `b`.

(The cleaner Statement 5 variant — `d ≥ d_S = d(A, B^T)`, the distance
of the block-erasure subsystem code `CSS(A, B^T)` — is also available
for the *fully abelian* Gross case, since Statement 9 / Statement 8
guarantee `δ_X = δ_Z = 0` automatically.)

  `G_a = ⟨(3,0), (0,1)⟩ = ⟨(3,0)⟩ × Z_6 = Z_4 × Z_6` (since (3,0) has

So the Tillich–Zémor bound's denominator is `c = 8` (or possibly 8 is

### Implication for the bound

If `d_A^⊥ ≥ K_A` and `d_B^⊥ ≥ K_B` and `c = 8`, then by Statement 12,
`d ≥ ⌈min(K_A, K_B) / 8⌉`.  Even an optimistic `K_A = K_B = 96` (the
empirical maximum classical group-code distance) gives `d ≥ 12`, hitting
the target.  But the realistic classical distance is much smaller —
likely in the range `[4, 20]`, giving `d ≥ 1` (trivial) or `d ≥ 2`
(useless).

**Therefore: the lifted-product bound is unlikely to deliver tight
`d ≥ 12` for the Gross code.**  This is a calibrated negative result.

## Approach decomposition

### B1 (preferred if `dim ker A` ≤ 20):

Compute `d(ker L(a))` directly via Lean's `native_decide` by enumerating
all elements of a `dim ker A`-dimensional F_2-space and finding the
minimum-weight non-zero one.  If `dim ker A = 12` (per the estimate),
this is `2^12 = 4096` evaluations of `conv a f = 0` followed by weight
computation.  Tractable.

**Risk**: enumeration may not actually be over the kernel — we'd have
to enumerate `f ∈ F_2^{72}`, filter to kernel.  That's `2^72`, infeasible.
A native_decide on the kernel would need a Lean-level representation of
the kernel as a free F_2-module of known dim, which is more work than
expected.

**Refined plan**: encode the kernel via a *basis* `(b_1, ..., b_k)` for
ker L(a), then enumerate `2^k` linear combinations.  This requires either:
(a) machine-computing the basis externally (Python/SageMath) and pasting
the coefficients into Lean, or (b) computing the basis inside Lean (via
`Matrix.kerLin` if it exists, or manual Gaussian elimination).
Option (a) is faster but is a "trusted external computation"; option (b)
is purer but takes hours of Lean engineering.

### B2 (more rigorous, less computational):

Identify a low-weight codeword in `ker L(a)` *analytically* and use it
to UPPER BOUND `d_A^⊥`.  Combined with Statement 12, this gives an
UPPER BOUND on the TZ bound — i.e., it tells us "TZ cannot give better
than `d ≥ K`".  This is the negative-result direction.

Candidate low-weight codewords in `ker L(a)`:
- Cofactor: if `(x^2 + x + 1) · (1 + y + y^2)` is in ker(L(a))?  Check:

## Stop criteria

- If Step 1 reveals `dim ker L(a) > 30` (much bigger than expected),
  abandon B1 and pivot to writing up negative result.
- If Step 1 yields `dim ker L(a) ≤ 16` AND a low-weight codeword
  (weight ≤ 8) can be exhibited, the TZ bound is at most `8/8 = 1`,
  trivially loose — write up as negative.
- If Step 1 yields a low-weight codeword of weight ≥ 96, the TZ bound
  hits the target; in that case, formalize aggressively (B3 budget
  becomes worth it).
- 8-hour session cap regardless.

## Probability estimates (calibrated)

- `d_0 ≥ 12` (tight): ~5%.  Requires `d_A^⊥ ≥ 96`, surprising for a

- `d_0 ≥ 4` (publishable partial): ~30%.
- `d_0 = 1` or `d_0 = 2` (effectively negative): ~50%.
