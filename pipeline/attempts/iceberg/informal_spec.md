# Informal spec: iceberg `[[2m, 2m−2, 2]]` error-detecting code

## Summary

The **iceberg code** family (a.k.a. **generalized parity code**, **self-dual
quantum parity-check code**, or "[[2m, 2m−2, 2]] detection code") is a
self-complementary, self-dual CSS stabilizer code on `n = 2m` physical
qubits, encoding `k = 2m − 2` logical qubits at distance `d = 2`. It detects
(but does not correct) any single-qubit Pauli error.

The stabilizer is generated by exactly **two operators**: the all-X
operator `XX...X` and the all-Z operator `ZZ...Z`, both acting on all `2m`
physical qubits. For `m = 2` this is the [[4, 2, 2]] four-qubit code; for
`m = 3` it is the [[6, 4, 2]] code; in general it is the **highest-rate
distance-2 CSS code** when an even number of qubits is used.

The "iceberg" name comes from the fact that each codeword `|b̄⟩` is a sum
of a "visible" basis state (an even-weight bitstring `b`) and an "invisible"
basis state (its bitwise negation `¬b`, which is also even-weight when `n`
is even).

## Parameters

- Physical qubits: **n = 2m** (parametric in `m`)
- Logical qubits: **k = 2m − 2**
- Distance: **d = 2** (detects a single error; cannot correct)
- Family: **CSS**, self-dual (the X-stab and Z-stab share the same support)
- Parameter constraint: **`m ≥ 2`**. The `m = 1` case would give a trivial
  [[2, 0, 2]] code with no logical qubits, which we exclude via
  `[Fact (2 ≤ m)]` (mirroring `RotatedSurfaceCodeN`'s `[Fact (3 ≤ L)]`).
- Originally introduced in:
  - [Steane 1996, `arxiv:quant-ph/9605021`] (general CSS framework)
  - [Gottesman 1997, `arxiv:quant-ph/9702029`] (stabilizer-formalism analysis)
  - [Viola–Knill–Lloyd 1998, `arxiv:quant-ph/9809071`] (dynamical decoupling)

**Source-availability note.** The arxiv abstracts retrieved during Stage 2
do not provide explicit per-logical-qubit logical-operator formulas; only the
two stabilizer generators are quoted directly from the EC Zoo entry. The
logical-operator convention chosen below is derived **by hand** from
first-principles symplectic commutation analysis, then cross-checked against
the existing `FourQubit_4_2_2.lean` formalization for the `m = 2`
specialization (see § "Relation to FourQubit_4_2_2.lean" below). It is
**not directly quoted** from any of the three references — the references
specify the codeword basis (and stabilizers) but leave the logical-operator
choice unspecified up to Clifford equivalence (any maximal set of weight-2
operators in the centralizer that pairwise (anti)commute correctly is a
valid logical basis).

## Stabilizer generators

Exactly **two generators**, `n − k = 2m − (2m − 2) = 2`:

```
S_X := X X X ... X   (X on every one of the 2m qubits, no I's)
S_Z := Z Z Z ... Z   (Z on every one of the 2m qubits, no I's)
```

In Lean notation:

```lean
def S_Z (m : ℕ) [Fact (2 ≤ m)] : NQubitPauliGroupElement (2 * m) :=
  ⟨0, NQubitPauliOperator.Z (2 * m)⟩

def S_X (m : ℕ) [Fact (2 ≤ m)] : NQubitPauliGroupElement (2 * m) :=
  ⟨0, NQubitPauliOperator.X (2 * m)⟩
```

These overlap at all `2m` qubit positions (every X-pair anticommutes
locally with the matching Z-pair). The count of anticommuting positions is
`2m`, which is even since `2m` is even — so `S_X` and `S_Z` commute. This
is the unique "iceberg constraint": the two full-support stabilizers
commute precisely because the number of qubits is even.

## Logical operators (k = 2m − 2 logicals)

We adopt the **split-anchor convention**:

- Qubits `0, 1, ..., 2m − 3` are the "**individual logical**" positions:
  logical qubit `i` (for `i : Fin (2m − 2)`) is anchored on physical qubit
  `i`.
- Qubit **`2m − 2`** is the "**Z-anchor**" (always part of every `Z̄_i`).
- Qubit **`2m − 1`** is the "**X-anchor**" (always part of every `X̄_i`).

For `i : Fin (2m − 2)`:

```
X̄_i := X_i · X_{2m-1}      (X at qubits i and 2m-1, I elsewhere; weight 2)
Z̄_i := Z_i · Z_{2m-2}      (Z at qubits i and 2m-2, I elsewhere; weight 2)
```

In Lean:

```lean
def logicalX (m : ℕ) [Fact (2 ≤ m)] (i : Fin (2 * m - 2)) :
    NQubitPauliGroupElement (2 * m) :=
  ⟨0,
    ((NQubitPauliOperator.identity (2 * m)).set ⟨i.val, _⟩ PauliOperator.X)
      |>.set ⟨2 * m - 1, _⟩ PauliOperator.X⟩

def logicalZ (m : ℕ) [Fact (2 ≤ m)] (i : Fin (2 * m - 2)) :
    NQubitPauliGroupElement (2 * m) :=
  ⟨0,
    ((NQubitPauliOperator.identity (2 * m)).set ⟨i.val, _⟩ PauliOperator.Z)
      |>.set ⟨2 * m - 2, _⟩ PauliOperator.Z⟩
```

(Index coercions `Fin (2m - 2) ↪ Fin (2m)` and explicit `Fin` constructions
for `2m - 1` / `2m - 2` are populated by the Lean skeleton; see
`gap_audit.md` for the `Fin` machinery used.)

### By-hand verification of the (anti)commutation table

For Pauli operators `P = ⟨0, p⟩` and `Q = ⟨0, q⟩` with phase 0, the
"number of qubits where they anticommute" is the cardinality of
`{i | (p i, q i) ∈ {(X,Z), (Z,X), (X,Y), (Y,X), (Y,Z), (Z,Y)}}`. They
commute iff this cardinality is even. (See
`NQubitPauliGroupElement.commutes_iff_even_anticommutes`.)

**Stabilizer commutation (sanity, already used to motivate `n = 2m`):**

| pair       | anticomm support | size | even? | result   |
|------------|------------------|------|-------|----------|
| S_X · S_Z  | all `2m` qubits  | `2m` | yes   | commute  |

**Logical–stabilizer commutation:**

| pair         | anticomm support       | size | even? | result   |
|--------------|------------------------|------|-------|----------|
| X̄_i · S_X    | ∅ (all X commute)      | 0    | yes   | commute  |
| X̄_i · S_Z    | `{i, 2m−1}`            | 2    | yes   | commute  |
| Z̄_i · S_X    | `{i, 2m−2}`            | 2    | yes   | commute  |
| Z̄_i · S_Z    | ∅ (all Z commute)      | 0    | yes   | commute  |

So every logical operator is in the centralizer of the stabilizer.

**Logical pairwise (anti)commutation:**

| pair               | anticomm support             | size | even? | result      |
|--------------------|------------------------------|------|-------|-------------|
| X̄_i · X̄_j         | ∅                            | 0    | yes   | commute     |
| Z̄_i · Z̄_j         | ∅                            | 0    | yes   | commute     |
| X̄_i · Z̄_i (i=j)   | `{i}` (qubit i has X vs Z)   | 1    | no    | anticommute |
| X̄_i · Z̄_j (i≠j)   | ∅ (supports are disjoint)*   | 0    | yes   | commute     |

(*) For `i ≠ j`, both indices are in `{0, ..., 2m−3}`, neither equals
`2m−2` nor `2m−1`. So `X̄_i`'s support `{i, 2m−1}` is disjoint from `Z̄_j`'s
support `{j, 2m−2}`: no anticommuting Pauli pair anywhere. Therefore commute.

The diagonal-anticommutation count of 1 (odd) for `X̄_i · Z̄_i` is the key
fact: the anticommuting position is precisely qubit `i` (since both
operators have non-identity Pauli there, and `X` anticommutes with `Z`),
while at the anchor positions only one operator has non-identity Pauli
(qubit `2m−1`: only X̄_i; qubit `2m−2`: only Z̄_i). Hence the anticommuting
set is exactly `{i}`, cardinality 1, odd ⇒ anticommute. ✓

### Why the X-anchor and Z-anchor are different qubits

If both anchors were the same qubit (say `2m−1`), then `X̄_i · Z̄_i` would
anticommute at both qubit `i` and qubit `2m−1` (count 2, even ⇒ commute) —
contradicting the required diagonal anticommutation. Splitting the anchors
across the two "extra" qubits gives the unique anticommuting position at
qubit `i`, satisfying the diagonal anticommutation.

### Why this differs from `FourQubit_4_2_2.lean` (m = 2 case)

For `m = 2` (n = 4, k = 2), our convention yields:

```
X̄_0 = X I I X  (qubits 0 and 3)
X̄_1 = I X I X  (qubits 1 and 3)
Z̄_0 = Z I Z I  (qubits 0 and 2)
Z̄_1 = I Z Z I  (qubits 1 and 2)
```

The existing `FourQubit_4_2_2.lean` formalization uses a **different**
logical basis (per its file header):

```
X̄_1 = I X I X   (qubits 1 and 3)
X̄_2 = I I X X   (qubits 2 and 3)
Z̄_1 = I I Z Z   (qubits 2 and 3)
Z̄_2 = I Z I Z   (qubits 1 and 3)
```

Both bases describe the **same code** (the same stabilizer subgroup, hence
the same code subspace), but the choice of logical-operator labels and
supports is different. They are equivalent under a Clifford on the logical
qubits — i.e., the two formalizations describe the same physical code but
with different "logical basis" choices.

**Per the agent spec, we do NOT prove equivalence here.** The parametric
iceberg family at `m = 2` and `FourQubit_4_2_2.lean` coexist as separate
formalizations, mirroring the `RepetitionCode3.lean` / `RepetitionCodeN.lean`
relationship (where `RepetitionCode3` is a special hand-tuned `n = 3`
instance and `RepetitionCodeN` is the parametric `n + 2` family; both prove
distance 1 and the resulting `StabilizerCode` objects are not lifted into
each other).

The two conventions are linked by the index relabeling `(i_iceberg, j_4_2_2)
= (0, 1) ↔ (1, 2)` plus a Clifford swap on the anchor structure; we leave
this lift as future work (research-track, not engineering-track).

## Codespace

The code subspace is the simultaneous +1 eigenspace of `S_X` and `S_Z`.
Following the EC Zoo description:

> Each codeword is a superposition of a computational basis state labeled
> by an even-weight bitstring `b` and a state labeled by the negation of
> `b`.

A logical basis state `|b̄⟩` (where `b ∈ {0,1}^(2m−2)` indexes the logical
register) can be written as

```
|b̄⟩ = (1/√2)( |b · c_Z(b) · c_X(b)⟩ + |¬b · ¬c_Z(b) · ¬c_X(b)⟩ )
```

where `c_Z(b) ∈ {0, 1}` is the parity of `b` (chosen so total bitstring has
even weight, making `S_Z` act as +1) and `c_X(b)` is determined by the
remaining even-weight constraint. The exact formula is not needed in the
formalization (it would only feed a "codespace dimension = 2^k" theorem,
which we do not prove directly — instead, the dimension follows from
`generators_independent` plus the `2^(n−k)` index theorem).

## Theorems to formalize

The skeleton's theorem set follows `_TEMPLATE.lean`'s §1–§14 layout,
parametrized over `m`.

### T1: Z-generator is Z-type
For every `g` in `{S_Z m}`, `IsZTypeElement g`. (Trivial: case-split on the
singleton, apply `IsZTypeElement` definition with `PauliOperator.IsZType`
holding at every qubit because `S_Z` is the all-Z operator.)
[Source: derived from the all-Z definition; conceptually trivial.]

### T2: X-generator is X-type
For every `g` in `{S_X m}`, `IsXTypeElement g`. (Trivial, mirrors T1.)

### T3: Cross-commutation `S_Z · S_X = S_X · S_Z`
The all-Z and all-X operators on `2m` qubits commute (anticomm count =
`2m`, even). The cleanest proof is `pauli_comm_even_anticommutes` followed
by reducing the filter to `Finset.univ` (count `2m`) and applying
`Nat.even_two_mul` or directly proving `Even (2 * m)`. Alternative: extract
the `commutes_of_pauli_inner_zero` style proof from
`symplecticInner_X_Z = (2m mod 2 = 0)`.
[Source: standard CSS commutation. The structurally cleanest form uses
`NQubitPauliOperator.symplecticInner` evaluation at the all-X / all-Z
operators on even `n`; analogue of `allX_allZ_anticommute` but with the
parity reversed.]

### T4: All generators commute pairwise
`∀ g h ∈ generators m, g * h = h * g`. Follows from T1, T2, T3 via
`CSSCommutationLemmas.ZType_commutes` / `XType_commutes`.

### T5: `−I ∉ subgroup`
The closure of `{S_X m, S_Z m}` does not contain the negative-identity
element. Standard CSS argument via
`CSS.negIdentity_not_mem_closure_union`, consuming T1, T2, T3.

### T6: Generator list / set equality
`listToSet generatorsList = generators m`. Direct unfolding via
`listToSet_cons` / `listToSet_nil`. The generator list is `[S_Z m, S_X m]`
(length `2 = n − k`).

### T7: Phase-zero generators
`AllPhaseZero generatorsList`. Trivial: both generators have phase 0.

### T8: Generator independence (linear independence of check-matrix rows)
The two check-matrix rows for `S_Z m` and `S_X m` are linearly independent
over `ZMod 2`. The check matrix of `[S_Z m, S_X m]` has shape `2 × 4m`; the
first row is `(0 ... 0, 1 ... 1)` (zeros on the X-part, ones on the
Z-part), and the second row is `(1 ... 1, 0 ... 0)`. These are independent
since neither row is a scalar multiple of the other (the only nonzero
combinations to consider are themselves, and `0 ≠ ones` on both halves).
**Parametric proof needed**: cannot `decide` because `m` is symbolic. Likely
unfolds into a finite case analysis on the `2 × 2` coefficient matrix in
`(ZMod 2)`.

### T9: Bundled `StabilizerGroup (2 * m)`
Standard `mkStabilizerFromGenerators` packaging, consuming T4 and T5.

### T10: `stabilizerGroup_toSubgroup_eq`
Bridge between `stabilizerGroup.toSubgroup` and the `Subgroup.closure
generators m` definition. Standard, mirrors `RepetitionCodeN`.

### T11: Logical X anticommutes Logical Z (diagonal)
For every `i : Fin (2m − 2)`,
`Anticommute (logicalX m i) (logicalZ m i)`. Anticomm support is `{i.val}`
in `Finset (Fin (2m))` — cardinality 1, odd. Use
`pauli_anticomm_odd_anticommutes` plus a parametric filter-equality proof.
This is the heart of the parametric proof — see plan.md §"Per-theorem
proof sketch / T11".

### T12: Logical operators commute pairwise (off-diagonal)
For every `i j : Fin (2m − 2)`:
- T12a: `logicalX m i · logicalX m j = logicalX m j · logicalX m i`
  (both X-type, trivial).
- T12b: `logicalZ m i · logicalZ m j = logicalZ m j · logicalZ m i`
  (both Z-type, trivial).
- T12c (i ≠ j): `logicalX m i · logicalZ m j = logicalZ m j · logicalX m i`.
  Anticomm support is empty (disjoint supports), so commute.

### T13: Logical X in centralizer (parametric)
For every `i : Fin (2m − 2)`, `logicalX m i ∈ centralizer (stabilizerGroup m)`.
Reduces via `Subgroup.forall_comm_closure_iff` to showing `logicalX m i`
commutes with each of the two generators `S_Z m` and `S_X m`:
- vs. `S_X m`: both X-type, commute (via `XType_commutes` or
  `pauli_comm_componentwise`).
- vs. `S_Z m`: anticomm at `{i.val, 2m - 1}`, count 2, even ⇒ commute.

### T14: Logical Z in centralizer (parametric)
For every `i : Fin (2m − 2)`, `logicalZ m i ∈ centralizer (stabilizerGroup m)`.
Symmetric to T13: vs. `S_Z m` both Z-type commute; vs. `S_X m` anticomm at
`{i.val, 2m - 2}`, count 2, even ⇒ commute.

### T15: `StabilizerCode (2 * m) (2 * m - 2)` packaging
Bundle T6–T14 plus the off-diagonal commutation T12 to produce the
`StabilizerCode` structure. The `logical_commute_cross` field uses the
"all four off-diagonal pairs commute" pattern from `FourQubit_4_2_2.lean`
generalized over the `Fin (2m − 2) × Fin (2m − 2)` index space.

### T16: `stabilizerCode_toSubgroup_eq`
Bridge for the distance proof: `stabilizerCode.toStabilizerGroup.toSubgroup
= Subgroup.closure (generators m)`. Standard `change` + unfold.

### T17: Weight-1 anticommute witness
The key lemma feeding the distance proof. For every qubit `i : Fin (2m)`
and every non-identity `PauliOperator P`, there exists `g ∈ generators m`
such that `Anticommute (weightOneAt i P) g`. The "easy" case:
- If `P = X` or `P = Y`: weight-1 element has non-Z at qubit `i`,
  anticommutes with the all-Z stabilizer `S_Z m` at qubit `i` (single
  anticomm position, odd ⇒ anticommute).
- If `P = Z`: weight-1 element has Z at qubit `i`, anticommutes with the
  all-X stabilizer `S_X m` at qubit `i` (single anticomm position, odd ⇒
  anticommute).

**No dichotomy needed** (unlike `CSS_4_1_2.lean`): every qubit is "covered"
by both `S_X` and `S_Z` (full-support stabilizers), so the witness choice
depends only on `P`, not on `i`.

### T18: `code_has_distance_two : HasCodeDistance stabilizerCode 2`
**Uses the PR #34 helper** `hasCodeDistance_two_of_anticommute_witness`:

```lean
theorem code_has_distance_two (m : ℕ) [Fact (2 ≤ m)] :
    HasCodeDistance (stabilizerCode m) 2 :=
  hasCodeDistance_two_of_anticommute_witness (stabilizerCode m)
    (generators m)
    (stabilizerCode_toSubgroup_eq m)
    (weight_one_anticomm_witness m)
    ⟨logicalX m ⟨0, ...⟩, ..., by ...⟩
```

The witness for the "exists weight-2 nontrivial logical" requirement is
`logicalX m 0` (which has weight exactly 2: X at qubits 0 and `2m - 1`).
`weight (logicalX m 0) = 2` requires a small parametric weight computation;
see `gap_audit.md` for the issue and the planned approach.

### T19: `StabilizerCodeWithDistance (2 * m) (2 * m - 2) 2` packaging
Trivial wrapper from T15 + T18.

## Edge cases / convention notes

1. **`m = 1` excluded.** Without `[Fact (2 ≤ m)]`, `k = 2m - 2 = 0` and the
   `StabilizerCode` would have zero logical qubits, making `Fin k = Fin 0`
   empty and the logical-operator universe vacuous. The theorems all still
   typecheck at `m = 1` but become vacuously true; we exclude this case to
   keep the `[[2m, 2m−2, 2]]` parameter table meaningful and to avoid
   `Nat.sub` underflow when computing `2m - 2` for `m = 1`.

2. **`2m - 2` and `2m - 1` use `Nat.sub`.** Since `m ≥ 2`, both `2m - 2 ≥ 2`
   and `2m - 1 ≥ 3`, so the subtraction is well-defined. We will need
   explicit `Fin` constructions `⟨2 * m - 1, by omega⟩` and
   `⟨2 * m - 2, by omega⟩` in the Lean code. The `[Fact (2 ≤ m)]` hypothesis
   discharges these bound proofs via `omega`. Mirror approach: see
   `RepetitionCode/N.lean`'s use of `Fin.castSucc` / `Fin.succ` for `n + 1`
   / `n + 2` indexing (different approach, but same flavor of "construct a
   `Fin` from a parameter computation").

3. **0-based qubit indexing throughout.** Matches the rest of the repo
   (`Steane7.lean`, `FourQubit_4_2_2.lean`, etc.).

4. **Phase conventions.** All generators and logical operators have
   `phasePower = 0`. No logical Y is defined in the skeleton (optional per
   `_TEMPLATE.lean §10`).

5. **Distance proof method: `hasCodeDistance_two_of_anticommute_witness`
   (PR #34).** This is the canonical closer for distance-2 CSS codes and
   was added to the framework specifically to avoid the
   `interval_cases w; rcases ...; exact` boilerplate that every distance-2
   formalization was instantiating verbatim. The witness function
   (T17 above) is straightforward because both stabilizers have full
   support — no need for the `hi_dichotomy` pattern from `CSS_4_1_2.lean`.

6. **Connection to existing repo abstractions:** the iceberg `[[2m, 2m−2, 2]]`
   parametric family is the **first generalized parametric CSS code with
   `k > 2` logical qubits**. The k-many off-diagonal logical commutations
   (T12) and the centralizer membership (T13, T14) are the first uses of
   parametric `Fin k` indexing for logicals in this repo. Test of the
   `LogicalQubitOps` API at scale.
